# Quickstart Reference

Describe the content that an effective quickstart should contain.

Before writing a quickstart, you MUST know:

- the minimum setup required for first success
- the shortest safe happy path
- the exact signal that confirms the flow worked

## Core Principles

- MUST optimize for the shortest safe path to first success
- MUST include only the steps required for that first useful result
- MUST make the success signal obvious so readers know when they are done
- NEVER turn the quickstart into a full tutorial
- NEVER hide required prerequisites or assumptions that would block success

## Quickstart Structure

Cover the following sections, including optional sections when relevant.

- Overview: what the reader will achieve by following the quickstart
- Audience `[Optional]`: who this quickstart is meant for and what starting point it assumes
- Prerequisites: only the minimum requirements needed to succeed
- Setup: the smallest viable setup sequence
- Run: the exact commands or actions to get the first result
- Success Check: how to confirm the quickstart worked
- Cleanup or Reset: optional teardown or recovery notes if relevant
- Expected Output `[Optional]`: the key output or result readers should see after running the flow
- Platform Notes `[Optional]`: small platform differences that affect setup or execution
- Known Limitations `[Optional]`: important shortcuts or limits of the quickstart path
- Next Steps: where to go for deeper usage, guides, or reference docs

## Next Steps

After the reader succeeds, point them to the README, a guide, or best-practices material for deeper understanding and repeated use.
