# README Reference

Describe the content that an effective README should contain.

Before writing a README, you MUST know:

- what the project or module provides
- the most common setup and usage paths
- which deeper documents should be linked instead of duplicated

## Core Principles

- MUST help a new reader understand quickly what this thing is and how to start using it
- MUST keep the README focused on orientation, setup, and common usage
- MUST link to deeper documents instead of duplicating everything in one file
- NEVER turn the README into a full handbook when separate documents already exist
- NEVER hide the main setup or usage path under too much background detail

## README Structure

Cover the following sections, including optional sections when relevant.

- Overview: what the project, package, module, or skill is and why it exists
- Audience `[Optional]`: who this README is mainly for and what they are likely trying to do
- What It Provides: the main capabilities or contents
- Structure: the important directories, files, or components readers should know
- Setup: how to install, configure, or prepare the environment
- Usage: the most common commands, workflows, or examples
- Development Workflow: how to contribute, test, or iterate locally
- Contribution Notes: how to contribute or where contribution rules live
- Status or Support Matrix `[Optional]`: version, platform, or support expectations when relevant
- Troubleshooting `[Optional]`: common setup or usage issues and how to resolve them
- FAQ `[Optional]`: repeated questions that are worth answering once
- Release Process `[Optional]`: how releases are created if readers need that context
- Related Documentation `[Optional]`: links to deeper guides, specs, or references

## Next Steps

After the README is written, check whether linked quickstarts, guides, specs, or contribution docs exist and are still aligned.
